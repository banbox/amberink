# Technical Constraints & Stack
1.  **核心业务逻辑 (High Fidelity)**：
    -   **文章即 NFT**：每篇文章发布时，本质上是铸造（Mint）了一个 NFT（或部署了一个相关合约），用户可以“收藏（Collect）”文章。
    -   **内容存储**：文章的具体内容（Markdown）和元数据（Metadata）存储在 Arweave 上。
2.  **区块链网络**：Optimism (L2)。
3.  **智能合约**：
    -   语言：Solidity。
    -   工具：Foundry。
    -   模式建议：请权衡“工厂模式（为每篇文章部署新合约，类似 Mirror v1）”与“单体 ERC-1155 模式（通过 TokenID 区分文章，更省 Gas）”的优劣，并推荐一种最适合当前 L2 环境的方案。
4.  **数据索引**：必须使用 **Subsquid SDK** 构建索引服务（Indexer），用于查询“某用户的所有文章”、“文章详情”等数据，前端不直接读取合约状态。
    -   需利用 Subsquid 的 **Archive** 和 **Processor** 架构优势。
5.  **前端框架**：**SvelteKit**。
    -   注意：鉴于 Web3 生态对 React 的偏向，请特别说明在 SvelteKit 中如何优雅地处理钱包连接（Wallet Connection）和合约交互（建议基于 `@wagmi/core` + `viem` 或 `svelte-wagmi` 等方案）。

要求还具有的逻辑：
用户可以关注某些作者，可以对文章进行评论；
文章应该具有分类属性，用户可以选择对某些分类感兴趣；
用户默认打开时，显示推荐的文章列表；推荐逻辑如下：
从感兴趣的分类文章、关注的作者新文章中，选择最热门（收藏或评论的数量）的5篇文章显示。
向下滚动时加载更多：展示剩余文章
当所有分类和关注文章都加载完毕时，加载所有分类下，最热门的文章；
注意上面所有阶段都应当进行去重，不显示用户已获取过的文章，或许可使用布隆过滤器？或SubSquid已有的内置机制实现。
